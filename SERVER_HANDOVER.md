# 服务器交接与重建说明

这份文档是写给下一位接手人员的。

最重要的一件事先说清楚：

- 当前线上服务器不是项目公共资产，而是上一任维护者个人购买和维护的。
- 接手后不要默认继续依赖这台旧服务器。
- 正确做法是：你们自己重新购买服务器、准备域名、完成备案与 HTTPS 配置，然后重新部署本项目。

这意味着你不能把“代码已上传 GitHub”理解成“线上环境已经交接完成”。  
代码只是项目内容，服务器、域名、证书、备案、解析和运维责任都需要你们重新接手或重建。

## 你们需要重新准备的东西

至少需要准备下面这些资源：

- 一台新的云服务器
- 一个你们自己控制的域名
- 域名备案能力
- 新域名的 DNS 解析权限
- 新服务器的 SSH 登录方式
- 新域名的 HTTPS 证书
- 项目部署目录、Python 环境、Nginx 和 systemd

如果这些资源还没准备好，不建议直接开始部署。

## 推荐的重建顺序

建议按这个顺序做，不容易乱：

1. 先买服务器
2. 再买域名
3. 做域名备案
4. 配置 DNS 解析到新服务器 IP
5. 申请 HTTPS 证书
6. 在新服务器部署项目
7. 配置 Nginx + systemd
8. 最后再做联调和上线

## 1. 购买新服务器

建议选择你们自己能长期续费、能稳定管理的云厂商账号，不要继续依赖个人私有机器。

选择服务器时至少注意：

- 操作系统建议用 Linux
- 需要有公网 IP
- 要能开放 `80` 和 `443` 端口
- 至少要支持 Python 运行环境
- 要确认你们自己保管服务器登录方式和续费权限

如果这是长期项目，建议从一开始就把“谁持有云账号、谁知道登录方式、谁负责续费”说清楚。

## 2. 购买域名

域名不要继续绑定在上一任维护者个人控制下。

建议：

- 新团队自己注册域名
- 确保域名控制权在接手团队手里
- 确保 DNS 解析权限也在接手团队手里

因为后续备案、证书申请、解析和迁移，都会依赖域名控制权。

## 3. 域名备案

如果你们使用的是中国大陆服务器，通常需要做备案后才能稳定正式上线。  
备案的具体流程、材料和时长会受云厂商和当时政策影响，所以要以你们实际购买服务器的云厂商要求为准。

重要提醒：

- 先确认服务器所在地是否需要备案
- 备案主体要明确，是学校、组织还是个人
- 域名实名信息、备案主体信息、服务器信息要保持一致或满足平台要求
- 备案可能需要时间，不要等到上线前一天才开始准备

如果备案没完成，很多情况下域名上线会受限。

## 4. 配置 DNS 解析

备案和服务器准备就绪后，需要把新域名解析到你们的新服务器公网 IP。

通常至少要处理：

- 主域名解析
- `www` 子域名解析

解析完成后，要确认：

- 域名已经指向新服务器
- 浏览器访问域名时，请求确实到达你们自己的机器

## 5. 申请 HTTPS 证书

旧项目里虽然有证书文件，但那不是给新团队长期复用的交接方式。  
你们应该为自己的新域名重新申请证书。

推荐做法：

- 使用免费 HTTPS 证书
- 用你们自己控制的新域名重新申请
- 把证书文件部署到你们自己的服务器上

重要提示：

- 证书必须和新域名匹配
- 证书有有效期，不是申请一次永久有效
- 需要提前考虑续期
- 不要把新的私钥和证书随便提交到公开仓库

如果你们后续使用的是自动续期方案，也要确认续期任务确实能跑通。

## 6. SSH 与服务器登录

接手团队必须建立自己的 SSH 管理方式，不要继续依赖上一任维护者的个人登录信息。

建议：

- 为新团队成员单独配置 SSH 登录
- 尽量不要多人共用同一套弱密码
- 优先使用更稳妥的登录方式
- 明确谁有服务器最高权限

最关键的是：

- 你们要能独立登录新服务器
- 你们要能独立重启服务
- 你们要能独立更换证书、改 Nginx、改部署目录

如果这些能力还依赖上一任维护者，就不算真正完成交接。

## 7. 新服务器上的部署内容

项目重建时，至少需要重新做这些事：

- 上传项目代码到新服务器
- 创建 Python 虚拟环境
- 安装 `requirements.txt`
- 配置 `data/admin_users.json`
- 准备 `data/` 和 `uploads/` 目录
- 配置 systemd
- 配置 Nginx
- 配置 HTTPS

当前仓库里已有这些参考文件：

- [DEPLOYMENT.md](/Users/doormi/Documents/time-test1/DEPLOYMENT.md)
- [deploy/nginx.conf](/Users/doormi/Documents/time-test1/deploy/nginx.conf)
- [deploy/volunteer-api.service](/Users/doormi/Documents/time-test1/deploy/volunteer-api.service)
- [deploy/time-test1.service](/Users/doormi/Documents/time-test1/deploy/time-test1.service)

但你们必须按自己的新环境重写或修改其中至少这些内容：

- 域名
- 证书路径
- 服务器用户
- 项目部署目录
- systemd 的 `WorkingDirectory`
- systemd 的 `ExecStart`
- Nginx 的 `server_name`
- Nginx 的 `root`

不要直接照搬旧配置就上线。

## 8. 上线前必须确认的事

正式上线前至少检查下面这些点：

- 域名已经解析到你们自己的服务器
- HTTPS 正常，浏览器没有证书错误
- `/health` 能返回正常结果
- 首页可以打开
- 学生查询可以正常请求后端
- 管理员可以登录
- 总表可以下载
- 上传功能可用
- systemd 能重启服务
- Nginx 重载后配置不报错

## 9. 和上一任环境彻底切开的建议

如果你们想避免以后再出现“服务器跟着个人走”的问题，建议一开始就把这些东西整理清楚：

- 云厂商账号由团队或组织统一持有
- 域名由团队或组织统一持有
- 备案主体明确
- SSH 管理方式明确
- 续费责任明确
- 证书续期责任明确
- 谁能改生产环境明确

## 10. 重要风险提醒

下面这些是非常容易在交接时踩坑的点：

- 不要默认旧服务器还能长期继续用
- 不要默认旧域名还能继续归你们控制
- 不要把旧证书文件当成长期方案
- 不要把服务器密码、私钥、证书继续直接提交到公开仓库
- 不要只拿到代码，却没有拿到云账号、域名控制权和部署能力

只拿到代码，不等于拿到了线上系统。  
真正完成交接的标准应该是：

- 你们自己有新的服务器
- 你们自己有新的域名
- 你们自己完成了备案
- 你们自己申请并部署了 HTTPS 证书
- 你们自己可以独立部署、重启、维护这个项目
